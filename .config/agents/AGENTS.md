# ガイドライン

このガイドラインでは、言語共通、言語別などに分け、一般に遵守すべき規則を示します。
すべてのケースにおいて、このガイドラインに記述されている限りではない場合がありますが、必ず以下の記述を考慮してください。

## 言語共通

- 型や関数は再利用を考慮して、同じようなコードを繰り返し記述しないようにしてください。
- 一つのファイル、モジュールには、ファイル名、モジュール名に即した内容のみを記述し、それぞれのコード量が肥大しないよう、適切に分割してください。
- コメントの言語はプロジェクトの現状に即したものにしてください。英語のコメントが多ければ英語、日本語が多ければ日本語といった具合です。
- 無駄にコメントを書く必要はありませんが、複雑で自明ではないコードへのコメントは記述してください。公開APIへのドキュメントコメントは、関数のシグネチャからわからない範囲のみ記述してください。
- テストにはこまめにコメントを記述することを求めます。
- 自明なテストの記述は避けてください。
- 非公開の関数に関するテストは、必要な場合のみにしてください。

## Language: Rust

- 基本的にはRustのベストプラクティスに従ってください。
- 手続き的な処理の記述を避け、イテレータや標準の関数を用いた記述を心がけてください。ただし、その記述が実行効率を損なう場合はこの限りではありません。
- `#[allow(...)]` の利用には慎重になってください。また、 `#![feature(...)]` といった、 unstable 機能の利用は、必ず確認をしてください。
- 新しいマクロの定義は慎重になってください。マクロの多用は可読性を下げますが、その方が良い場合もあります。
- `unsafe` や `Box::leak` 、 `forget` は一般に使用するべきではありません。多少の最適化のために使うのは推奨しません。もし使用する場合、確認をとってください。

### API設計

- 綺麗なAPI設計を心がけ、外部のユーザが使えるように、CLI部分以外は `src/lib.rs` として実装してください。
- APIの公開、非公開については熟慮してください。なんでも公開にしないようにしてください。
- バイナリクレートは、 `src/bin/` 以下に作成してください。

### 依存クレート

- 依存関係追加時は、必ず crates.io を参照し、最新のバージョンを用いてください。
- 適切な粒度のバージョン指定を心がけてください。

### 型の設計

- 特定用途の値は、専用の型を作成する new type pattern を採用してください。
- 定義した型に無駄に多くのトレイトを実装する必要はありませんが、`Clone`, `Debug` などを `#[derive(...)]` を用いて実装してください。可能な場合は `Copy`, `Eq`, `Default` なども実装すると良いでしょう。
- 関数の引数、返り値の型は適切にトレイトなどを用い、また新しく定義する型もジェネリックな設計を心がけてください。
- メンテナンスを考え、重複したコードは避けてください。トレイトを用いて共通化してください。

### コード品質

- コード品質の維持にClippyを用いています。作業の区切りの際に `cargo clippy --all-targets` を積極的に利用してください。
- すべて書き終えたら、 `cargo fmt` をしてフォーマットを整えてください。

### 特定のクレートに関する規約

- serdeクレートを用いる場合は、公開する型では `Serialize`, `Deserialize` の両方を実装した方がいい場合が多くあります。また、 `#[serde(rename_all = "snake_case")]` や、 `#[serde(tag = "...")]` などを積極的に活用してください。


## Language: Python

- 基本的にはPythonのベストプラクティスに従ってください。
- パッケージマネージャにuvを用いて、スクリプトの実行には `uv run` を用いてください。
- コードのフォーマットはblackを用いて、書き終えた際にフォーマットを実行してください。
- パッケージの追加は、 `pypi.org` などで最新のパッケージを調べて、依存関係が解決可能な範囲で最新のものを利用してください。
- こまめなコメントを心がけてください。

### 型注釈

- 基本的にすべての関数に型注釈をつけてください。
- 定期的に `ty check` を用いて型検査してください。
- `import typing` はできるだけ避け、 `list` などの標準の型注釈を使ってください。
